# Clinical Monitoring ETL Pipeline

A portfolio project demonstrating an end-to-end ETL pipeline applied to synthetic clinical trial Site Visit Reports (SVRs). Built to showcase data engineering and healthcare analytics skills.

## What It Does

```
Extract  →  Parse findings from Site Visit Report PDFs (pdfplumber)
Transform →  Classify findings by category, severity, and status
             (keyword rules + Anthropic Claude LLM fallback)
Load     →  Write structured CSVs ready for Power BI / Tableau
```

## Project Structure

```
clinical-monitoring-etl/
├── data/
│   ├── raw/                   # Synthetic SVR PDFs (5 reports, 16 findings)
│   └── processed/
│       ├── findings.csv       # One row per finding, fully classified
│       └── summary.csv        # Rollup by report: counts by severity & status
├── generate_pdfs.py           # Generates synthetic SVR PDFs via reportlab
├── etl_pipeline.py            # Extract → classify → load
├── config.py                  # Keyword rule sets and output column schema
├── requirements.txt
└── README.md
```

## Key Design Decisions

### Two-Stage Classification

1. **Keyword rules** (config.py): fast, deterministic, zero API cost. Covers 7 finding categories with curated clinical trial vocabulary (protocol deviation, informed consent, eCRF, adverse events, IP accountability, regulatory docs, equipment).
1. **LLM fallback** (Anthropic Claude Sonnet): fires only when keyword rules return no match. Category classification prompt returns structured JSON; severity and status are always sourced from the authoritative PDF marker line — not inferred.

### HIPAA-Safe Synthetic Data

All PDFs are entirely fictional. Sponsors, sites, monitors, subjects, and finding text are generated from hardcoded pools — no real patient data, no PHI. Safe to publish publicly on GitHub.

### Output Schema

`findings.csv` columns:

|Column                 |Description                              |
|-----------------------|-----------------------------------------|
|`report_id`            |SVR identifier (e.g. SVR-2025-0041)      |
|`site_id`              |Site identifier (e.g. SITE-001)          |
|`sponsor`              |Sponsoring organization                  |
|`protocol_number`      |Study protocol ID                        |
|`visit_date`           |Date of monitoring visit                 |
|`monitor_name`         |CRA name and credential                  |
|`finding_id`           |Unique finding key (report_id + sequence)|
|`finding_text`         |Full finding description                 |
|`category`             |Classified finding type                  |
|`severity`             |Critical / Major / Minor                 |
|`status`               |Open / In Progress / Closed              |
|`classification_method`|`keyword` or `llm`                       |

## Setup & Usage

```bash
# Install dependencies
pip install -r requirements.txt

# Generate synthetic PDFs (run once)
python generate_pdfs.py

# Run the ETL pipeline (keyword-only mode, no API key needed)
python etl_pipeline.py

# Run with LLM fallback enabled
ANTHROPIC_API_KEY=your_key_here python etl_pipeline.py
```

Processed CSVs are written to `data/processed/`. Connect Power BI or Tableau directly to these files for visualization.

## Background

Site Visit Reports are generated by Clinical Research Associates (CRAs) during monitoring visits to clinical trial sites. They document protocol deviations, data quality issues, regulatory gaps, and safety reporting concerns. Automating extraction and classification of findings supports risk-based monitoring (RBM) analytics and trial quality oversight.
